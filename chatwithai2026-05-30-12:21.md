Lesson 8: useContext

Remember "lifting state up" from Lesson 5? That works fine for one or two levels. But imagine a deeply nested component tree where a value like the logged-in user needs to reach a component 5 levels deep. You'd have to pass it as a prop through every level in between — even components that don't use it. That's called prop drilling, and it gets messy fast.

useContext solves this by creating a "broadcast channel" that any component in the tree can tune into directly.

Three steps to use it:

1. Create the context:


import { createContext } from 'react';

const UserContext = createContext(null);
2. Wrap your tree with a Provider and give it a value:


function App() {
  const [user, setUser] = useState({ name: 'Alice' });

  return (
    <UserContext.Provider value={user}>
      <Dashboard />
    </UserContext.Provider>
  );
}
3. Consume it anywhere inside that tree:


import { useContext } from 'react';

function Avatar() {
  const user = useContext(UserContext);
  return <img alt={user.name} />;
}
Avatar can be 10 levels deep inside Dashboard — it doesn't matter. It reaches the context directly, no props needed in between.

When to use it vs. props:

Props: when the data is only needed 1-2 levels down — keep it simple
Context: when the data is truly "global" to a subtree — current user, theme, language, auth state
Context is not a replacement for all props. Overusing it makes components harder to reuse because they depend on an implicit global instead of explicit inputs.

Lesson 9: Custom Hooks

You've seen useState, useEffect, and useContext. React lets you build your own hooks by combining them — this is how you extract and reuse logic across components.

A custom hook is just a function whose name starts with use and that calls other hooks inside it.

Example — extracting a data-fetching pattern:

Instead of repeating this in every component that fetches data:


function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    setLoading(true);
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => { setUser(data); setLoading(false); });
  }, [userId]);

  if (loading) return <p>Loading...</p>;
  return <h1>{user.name}</h1>;
}
You extract it into a custom hook:


function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    setLoading(true);
    fetch(url)
      .then(res => res.json())
      .then(result => { setData(result); setLoading(false); });
  }, [url]);

  return { data, loading };
}
Then use it anywhere cleanly:


function UserProfile({ userId }) {
  const { data: user, loading } = useFetch(`/api/users/${userId}`);

  if (loading) return <p>Loading...</p>;
  return <h1>{user.name}</h1>;
}

function ProductPage({ productId }) {
  const { data: product, loading } = useFetch(`/api/products/${productId}`);
  // ...
}
The key insight: custom hooks are just the Node.js module pattern applied to stateful logic. Same idea as extracting a service or utility module — except here you're extracting hooks that carry their own state and side effects.

Lesson 10: useReducer

useState works well for simple, independent values. But when state has multiple sub-values that change together, or when the next state depends on the previous one in complex ways, useReducer is a cleaner fit.

It's the same pattern as Redux — but built into React, no library needed.


import { useReducer } from 'react';

const initialState = { count: 0, step: 1 };

function reducer(state, action) {
  switch (action.type) {
    case 'increment':
      return { ...state, count: state.count + state.step };
    case 'decrement':
      return { ...state, count: state.count - state.step };
    case 'setStep':
      return { ...state, step: action.payload };
    case 'reset':
      return initialState;
    default:
      throw new Error(`Unknown action: ${action.type}`);
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, initialState);

  return (
    <div>
      <p>Count: {state.count} (step: {state.step})</p>
      <button onClick={() => dispatch({ type: 'increment' })}>+</button>
      <button onClick={() => dispatch({ type: 'decrement' })}>-</button>
      <button onClick={() => dispatch({ type: 'setStep', payload: 5 })}>Step = 5</button>
      <button onClick={() => dispatch({ type: 'reset' })}>Reset</button>
    </div>
  );
}
The mechanics:

reducer(state, action) — a pure function: takes current state + an action, returns next state. Never mutates state directly
dispatch(action) — sends an action to the reducer, triggers re-render
The reducer is completely separate from the component — easy to test in isolation
When to choose useReducer over useState:

Situation	Use
1-2 independent simple values	useState
Multiple values that change together	useReducer
Next state depends on previous state	useReducer
Complex business logic in state transitions	useReducer
You want to unit-test state logic separately	useReducer
From your Node.js background — this is exactly the event sourcing pattern. An action is an event, the reducer is the handler, and the state is what gets updated. If you've ever worked with a message queue or command pattern, this will feel very natural.

Example 1: Form with multiple fields

With useState you end up with a mess:


const [name, setName] = useState('');
const [email, setEmail] = useState('');
const [password, setPassword] = useState('');
const [loading, setLoading] = useState(false);
const [error, setError] = useState(null);
With useReducer, one state object, one place to reason about it:


const initialState = { name: '', email: '', password: '', loading: false, error: null };

function reducer(state, action) {
  switch (action.type) {
    case 'setField':
      return { ...state, [action.field]: action.value };
    case 'submit':
      return { ...state, loading: true, error: null };
    case 'success':
      return { ...initialState }; // reset form
    case 'error':
      return { ...state, loading: false, error: action.message };
  }
}

function SignupForm() {
  const [state, dispatch] = useReducer(reducer, initialState);

  const handleSubmit = async () => {
    dispatch({ type: 'submit' });
    try {
      await api.signup(state);
      dispatch({ type: 'success' });
    } catch (e) {
      dispatch({ type: 'error', message: e.message });
    }
  };

  return (
    <form>
      <input value={state.name} onChange={e => dispatch({ type: 'setField', field: 'name', value: e.target.value })} />
      <input value={state.email} onChange={e => dispatch({ type: 'setField', field: 'email', value: e.target.value })} />
      {state.error && <p>{state.error}</p>}
      <button onClick={handleSubmit} disabled={state.loading}>
        {state.loading ? 'Saving...' : 'Sign up'}
      </button>
    </form>
  );
}
Problem solved: State transitions that need to be atomic. When you submit, you want loading: true AND error: null to happen together in one render — not two separate setState calls that could cause two renders or a flash of stale error message.

Example 2: Shopping cart


function reducer(state, action) {
  switch (action.type) {
    case 'addItem': {
      const exists = state.items.find(i => i.id === action.item.id);
      if (exists) {
        return {
          ...state,
          items: state.items.map(i =>
            i.id === action.item.id ? { ...i, qty: i.qty + 1 } : i
          )
        };
      }
      return { ...state, items: [...state.items, { ...action.item, qty: 1 }] };
    }
    case 'removeItem':
      return { ...state, items: state.items.filter(i => i.id !== action.id) };
    case 'clearCart':
      return { ...state, items: [] };
  }
}
Problem solved: Business logic lives in the reducer, not scattered across components. Any component that dispatches addItem gets the same behavior — the "if it already exists, increment qty" logic is in one place.

Example 3: Multi-step wizard / checkout flow


const initialState = { step: 1, shippingInfo: null, paymentInfo: null, orderId: null };

function reducer(state, action) {
  switch (action.type) {
    case 'nextStep':
      return { ...state, step: state.step + 1 };
    case 'prevStep':
      return { ...state, step: state.step - 1 };
    case 'saveShipping':
      return { ...state, shippingInfo: action.data, step: 2 };
    case 'savePayment':
      return { ...state, paymentInfo: action.data, step: 3 };
    case 'orderPlaced':
      return { ...state, orderId: action.orderId, step: 4 };
  }
}
Problem solved: The entire flow state is in one place. Any step component only dispatches actions — it doesn't need to know what step comes next or what other steps have stored. You can also serialize the whole state and restore a user's progress.

The pattern behind all three examples:

Problem	Why useReducer wins
Multiple values that must change atomically	One dispatch = one render
Business logic growing inside components	Reducer is pure, testable in isolation
State transitions that depend on previous state	Reducer always receives the current state
Debugging — "how did we get into this state?"	You can log every action and replay them
That last point is big — because actions are plain objects, you can log every dispatch and build a complete history of what happened. Redux DevTools works on exactly this principle.
