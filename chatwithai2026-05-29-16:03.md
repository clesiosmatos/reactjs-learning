Lesson 1: What React actually is

You know Node.js — it runs JavaScript on the server and lets you build things by composing modules together. React is the same idea, but for the browser UI.

The core idea is simple: instead of manually manipulating the DOM (like with document.querySelector and element.innerHTML), you describe what the UI should look like, and React handles updating the DOM for you.

Think of it like this:

Vanilla JS: "Find that element, change this text, add that class" — imperative
React: "Here's what the UI looks like when data is X" — declarative
React lets you build UIs as a tree of components — each component is just a JavaScript function that returns HTML-like markup (called JSX).

A minimal React component looks like this:


function Greeting() {
  return <h1>Hello, world!</h1>;
}
That's it. A function that returns markup. React calls it, gets the output, and puts it in the DOM.

Lesson 2: JSX

That HTML-looking syntax inside the JavaScript function is called JSX. It's not real HTML — it gets compiled to plain JavaScript before the browser sees it. But you write it like HTML, which makes it readable.

A few key rules that differ from HTML:

1. class becomes className — because class is a reserved word in JS:


<div className="container">Hello</div>
2. Self-closing tags must close — unlike HTML where <input> is fine, in JSX you need:


<input />
<br />
3. You can embed JavaScript expressions with {}:


function Greeting() {
  const name = "Alice";
  return <h1>Hello, {name}!</h1>;
}
Anything inside {} is evaluated as JavaScript. So {2 + 2} renders 4, {name.toUpperCase()} renders ALICE, etc.

4. A component must return a single root element. This is invalid:


// WRONG - two root elements
return (
  <h1>Title</h1>
  <p>Text</p>
);
You wrap them in a <div> or an empty <> (called a Fragment):


return (
  <>
    <h1>Title</h1>
    <p>Text</p>
  </>
);

Lesson 3: Props

Components would be useless if they always rendered the same thing. Props (short for properties) are how you pass data into a component — exactly like passing arguments to a function.


function Greeting(props) {
  return <h1>Hello, {props.name}!</h1>;
}

// Using it:
<Greeting name="Alice" />
<Greeting name="Bob" />
In practice, people almost always destructure props directly:


function Greeting({ name }) {
  return <h1>Hello, {name}!</h1>;
}
You can pass any JavaScript value as a prop — strings, numbers, arrays, objects, even functions. Strings use quotes, everything else uses {}:


<User name="Alice" age={30} active={true} scores={[10, 20, 30]} />
One important rule: props are read-only. A component must never modify its own props. This is a hard rule in React — the data flows one way, from parent to child. If you need to change something, that's where state comes in (next lesson).

Think of it like a pure function — same props in, same output every time.

Lesson 4: State

Props are data passed into a component. State is data that lives inside a component and can change over time. When state changes, React automatically re-renders the component with the new data.

You create state with the useState hook:


import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}
Breaking it down:

useState(0) — creates a state variable with initial value 0
It returns an array with two things: the current value (count) and a setter function (setCount)
When you call setCount(newValue), React updates the value and re-renders the component
The critical rule: never mutate state directly. Always use the setter:


// WRONG
count = count + 1;

// CORRECT
setCount(count + 1);
This is how React knows something changed and needs to re-render. If you mutate directly, React has no idea.

Coming from Node.js — think of it like an event emitter. Calling setCount is like emitting an event that says "this data changed, please update the UI."

Question 6: Does the useState pattern make sense? Any questions about when state lives in one component vs another?

After answer this question, why we call useState as hook, isn't this a function?
It is a function — you're right. The name "hook" is a React-specific convention, not a different language construct.

React calls them hooks because they let you hook into React's internal machinery from a plain function component. When you call useState, you're not just declaring a variable — you're reaching into React's rendering engine and saying "track this value for me across re-renders."

The reason this matters: React needs to remember the value of count between renders. A normal local variable inside a function would be thrown away every time the function runs. React maintains a hidden data structure behind the scenes that persists state — useState is your hook into that structure.

All hooks follow the same pattern — they're functions that tap into React internals:

useState — hooks into React's state storage
useEffect — hooks into React's lifecycle (we'll cover this soon)
useContext — hooks into React's context system
There's also one important rule that exists because they hook into internals: hooks must always be called in the same order, at the top level of your component. Never inside if blocks or loops. React tracks hooks by their call order, so changing that order breaks things.


// WRONG
if (someCondition) {
  const [value, setValue] = useState(0); // breaks React's tracking
}

// CORRECT
const [value, setValue] = useState(0); // always called, unconditionally

Good catch — let me clarify that.

In React, every time state changes, React calls your component function again from scratch to get the new UI. So this happens:


function Counter() {
  let count = 0; // reset to 0 EVERY time React calls this function

  return (
    <button onClick={() => { count = count + 1; }}>
      Count: {count}
    </button>
  );
}
Click the button — count becomes 1. But then React re-renders, calls Counter() again, and let count = 0 runs again. The value is gone.

With useState, React stores the value outside your function, in its own internal memory:


function Counter() {
  const [count, setCount] = useState(0); // React fetches the stored value, not 0
  // ...
}
The 0 in useState(0) is only used the first time. After that, React ignores it and returns whatever value it has stored internally. That value survives across re-renders.

A simple analogy from Node.js: it's like the difference between a local variable inside a request handler (dies after the request) vs something stored in a database (persists across requests). useState is React's way of persisting data across renders.
