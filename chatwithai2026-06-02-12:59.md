Lesson 19: TypeScript + React
You already know TypeScript basics from Node. The React-specific parts are: typing props, hooks, events, and API responses. That's the entire lesson.

Props
The most common thing you'll type. Just an interface:


interface ButtonProps {
  label: string;
  onClick: () => void;
  variant?: 'primary' | 'secondary'; // optional with union
  disabled?: boolean;
}

function Button({ label, onClick, variant = 'primary', disabled }: ButtonProps) {
  return (
    <button onClick={onClick} disabled={disabled} className={variant}>
      {label}
    </button>
  );
}
For components that accept children:


interface CardProps {
  title: string;
  children: React.ReactNode; // anything React can render
}

function Card({ title, children }: CardProps) {
  return (
    <div>
      <h2>{title}</h2>
      {children}
    </div>
  );
}
React.ReactNode is the catch-all for children — string, number, JSX, arrays, null, undefined.

useState
TypeScript infers the type from the initial value when it can:


const [count, setCount] = useState(0);        // inferred: number
const [name, setName] = useState('');          // inferred: string
const [active, setActive] = useState(false);   // inferred: boolean
When the initial value is null or the type can't be inferred, annotate explicitly:


const [user, setUser] = useState<User | null>(null);
const [items, setItems] = useState<string[]>([]);
useRef
Two different uses, two different types:


// Referencing a DOM element — initial value must be null
const inputRef = useRef<HTMLInputElement>(null);

// Mutable value (like an instance variable) — no null needed
const timerRef = useRef<ReturnType<typeof setTimeout>>(undefined);

// Usage
inputRef.current?.focus(); // optional chain because it starts null
HTMLInputElement, HTMLButtonElement, HTMLDivElement — the pattern is HTML{Tag}Element.

Event Handlers
This trips up almost everyone. Event types come from React, not the DOM:


function SearchInput() {
  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    console.log(e.target.value);
  };

  const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault();
  };

  const handleClick = (e: React.MouseEvent<HTMLButtonElement>) => {
    console.log(e.currentTarget);
  };

  const handleKeyDown = (e: React.KeyboardEvent<HTMLInputElement>) => {
    if (e.key === 'Enter') { ... }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input onChange={handleChange} onKeyDown={handleKeyDown} />
      <button onClick={handleClick}>Search</button>
    </form>
  );
}
The pattern: React.{EventType}Event<HTML{Element}Element>.

For inline handlers, TypeScript infers the type automatically — you only need the explicit annotation when extracting to a named function:


// No annotation needed — inferred
<input onChange={(e) => setValue(e.target.value)} />
useReducer
Type the state and action:


type State = {
  count: number;
  error: string | null;
};

type Action =
  | { type: 'increment' }
  | { type: 'decrement' }
  | { type: 'reset'; payload: number };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'increment': return { ...state, count: state.count + 1 };
    case 'decrement': return { ...state, count: state.count - 1 };
    case 'reset':     return { ...state, count: action.payload }; // payload only available here
  }
}

const [state, dispatch] = useReducer(reducer, { count: 0, error: null });
The discriminated union on Action gives you type narrowing inside switch — TypeScript knows action.payload only exists on the reset case.

useContext
Type the context at creation:


interface AuthContextType {
  user: User | null;
  login: (email: string, password: string) => Promise<void>;
  logout: () => void;
}

const AuthContext = createContext<AuthContextType | null>(null);

// Custom hook that handles the null check once
function useAuth(): AuthContextType {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth must be used within AuthProvider');
  return ctx;
}
The custom hook pattern is important — it removes the | null from every usage site and gives a useful error message if someone uses it outside the provider.

API Responses (React Query)
Define your types, pass them as generics:


interface User {
  id: number;
  name: string;
  email: string;
}

// useQuery<TData, TError>
const { data, error } = useQuery<User, Error>({
  queryKey: ['user', userId],
  queryFn: (): Promise<User> => fetch(`/api/users/${userId}`).then(res => res.json()),
});

// data is User | undefined  (undefined while loading)
// error is Error | null
Or infer it from the queryFn return type — if queryFn returns Promise<User>, data is already typed as User | undefined:


async function fetchUser(id: number): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}

const { data } = useQuery({
  queryKey: ['user', userId],
  queryFn: () => fetchUser(userId), // data inferred as User | undefined
});
Component Return Types
You rarely need to annotate these — TypeScript infers JSX.Element automatically. The one case where it matters is when you might return null:


// TypeScript infers this fine
function Button({ label }: ButtonProps) {
  return <button>{label}</button>;
}

// Explicit annotation when the function signature needs to be clear
function MaybeRender({ show }: { show: boolean }): JSX.Element | null {
  if (!show) return null;
  return <div>Visible</div>;
}
Generics in components
Useful for list-type components:


interface ListProps<T> {
  items: T[];
  renderItem: (item: T) => React.ReactNode;
  keyExtractor: (item: T) => string;
}

function List<T>({ items, renderItem, keyExtractor }: ListProps<T>) {
  return (
    <ul>
      {items.map((item) => (
        <li key={keyExtractor(item)}>{renderItem(item)}</li>
      ))}
    </ul>
  );
}

// Usage — T inferred as User
<List
  items={users}
  keyExtractor={(u) => u.id.toString()}
  renderItem={(u) => <span>{u.name}</span>}
/>

Thing	Type

Lesson 20: Next.js — SSR, SSG, and the App Router
What Next.js is and why it exists
React is a UI library — it runs in the browser. Next.js is a framework built on top of React that adds:

File-based routing — no React Router setup
Server-side rendering (SSR) — render HTML on the server per request
Static generation (SSG) — render HTML at build time
API routes — backend endpoints in the same project
Server Components — components that run only on the server (no JS sent to browser)
The mental shift: in plain React, everything runs in the browser. In Next.js, code can run on the server, at build time, or in the browser — and you control which.

Two routers: Pages vs App
Next.js has two routing systems. You'll encounter both in the wild:

Pages Router (/pages)	App Router (/app)
Introduced	Next.js v1	Next.js v13 (stable v14)
Status	Legacy, still supported	Current standard
Components	All are Client Components	Default is Server Components
Data fetching	getServerSideProps, getStaticProps	async components, fetch
Learn the App Router — it's the future. But recognize Pages Router patterns when you see them in older codebases.

File-based routing (App Router)
The folder structure under /app defines your routes:


app/
├── layout.tsx          → wraps every page (persistent shell)
├── page.tsx            → /
├── about/
│   └── page.tsx        → /about
├── blog/
│   ├── page.tsx        → /blog
│   └── [slug]/
│       └── page.tsx    → /blog/anything  (dynamic route)
└── dashboard/
    ├── layout.tsx      → layout only for /dashboard/*
    ├── page.tsx        → /dashboard
    └── settings/
        └── page.tsx    → /dashboard/settings
page.tsx = the route. layout.tsx = persistent wrapper. [param] = dynamic segment.

Server Components vs Client Components
This is the core mental model shift.

Server Components (default in App Router):

Render on the server — HTML sent to browser
Can be async — await data directly, no useEffect
Zero JavaScript sent to the browser for these components
Cannot use useState, useEffect, event handlers, or browser APIs
Client Components — opt in with 'use client':

Render in the browser (and also SSR'd for the initial HTML)
Can use all hooks, event handlers, browser APIs
JS bundle is sent to the browser

// app/users/page.tsx — Server Component (no 'use client')
// This runs on the server. No useEffect, no loading state needed.
async function UsersPage() {
  const users = await fetch('https://api.example.com/users').then(r => r.json());

  return (
    <ul>
      {users.map(u => <li key={u.id}>{u.name}</li>)}
    </ul>
  );
}

export default UsersPage;

// components/SearchInput.tsx — Client Component
'use client';

import { useState } from 'react';

export function SearchInput({ onSearch }: { onSearch: (q: string) => void }) {
  const [query, setQuery] = useState('');

  return (
    <input
      value={query}
      onChange={e => setQuery(e.target.value)}
      onKeyDown={e => e.key === 'Enter' && onSearch(query)}
    />
  );
}
Rule of thumb: push 'use client' as far down the tree as possible. Keep data fetching and static rendering in Server Components; isolate interactivity to leaf components.

Data fetching in the App Router
No more useEffect + useState for server data. Server Components can just await:


// app/blog/[slug]/page.tsx
interface Props {
  params: { slug: string };
}

async function BlogPost({ params }: Props) {
  const post = await fetch(`https://api.example.com/posts/${params.slug}`)
    .then(r => r.json());

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
    </article>
  );
}

export default BlogPost;
Next.js extends fetch with caching options:


// Cache forever (like SSG) — regenerate only on deploy
const data = await fetch(url, { cache: 'force-cache' });

// Never cache (like SSR) — fresh on every request
const data = await fetch(url, { cache: 'no-store' });

// Revalidate every N seconds (ISR — Incremental Static Regeneration)
const data = await fetch(url, { next: { revalidate: 60 } });
Think of it like HTTP cache headers — you're telling Next.js how fresh the data needs to be.

Layouts
layout.tsx wraps child routes and persists across navigation — the shell doesn't re-render when you navigate:


// app/layout.tsx — root layout (required)
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <nav>...</nav>
        {children}  {/* page content renders here */}
        <footer>...</footer>
      </body>
    </html>
  );
}
Nested layouts work the same way — app/dashboard/layout.tsx wraps everything under /dashboard:


// app/dashboard/layout.tsx
export default function DashboardLayout({ children }: { children: React.ReactNode }) {
  return (
    <div className="dashboard">
      <Sidebar />
      <main>{children}</main>
    </div>
  );
}
Dynamic routes and generateStaticParams
[slug] catches any value. For SSG, tell Next.js which values to pre-render:


// app/blog/[slug]/page.tsx

// Runs at build time — generates /blog/post-1, /blog/post-2, etc.
export async function generateStaticParams() {
  const posts = await fetch('https://api.example.com/posts').then(r => r.json());
  return posts.map(post => ({ slug: post.slug }));
}

// Runs at build time for each slug
async function BlogPost({ params }: { params: { slug: string } }) {
  const post = await fetch(`https://api.example.com/posts/${params.slug}`)
    .then(r => r.json());
  return <article>{post.title}</article>;
}
Without generateStaticParams, the page renders on-demand (SSR). With it, pages are pre-built at deploy time (SSG).

API Routes
Put backend endpoints alongside your frontend — no separate Express server needed for simple cases:


// app/api/users/route.ts
import { NextRequest, NextResponse } from 'next/server';

export async function GET(request: NextRequest) {
  const users = await db.user.findMany();
  return NextResponse.json(users);
}

export async function POST(request: NextRequest) {
  const body = await request.json();
  const user = await db.user.create({ data: body });
  return NextResponse.json(user, { status: 201 });
}
GET, POST, PUT, DELETE — named exports map to HTTP methods.

loading.tsx and error.tsx
Next.js has built-in file conventions for Suspense and error boundaries:


app/dashboard/
├── page.tsx
├── loading.tsx   → shown while page.tsx is loading (Suspense fallback)
└── error.tsx     → shown if page.tsx throws (Error Boundary)

// app/dashboard/loading.tsx
export default function Loading() {
  return <Spinner />;
}

// app/dashboard/error.tsx
'use client'; // error boundaries must be client components

export default function Error({ error, reset }: { error: Error; reset: () => void }) {
  return (
    <div>
      <p>{error.message}</p>
      <button onClick={reset}>Retry</button>
    </div>
  );
}
No manual <Suspense> or <ErrorBoundary> wrappers needed — the file convention handles it.

The full mental model

Request hits /dashboard/settings
       ↓
app/layout.tsx          ← Server Component, renders once
app/dashboard/layout.tsx ← Server Component, renders once
app/dashboard/settings/page.tsx ← async Server Component, fetches data
       ↓
HTML sent to browser (fast first paint)
       ↓
'use client' components hydrate (attach event listeners)
The browser gets HTML immediately — no blank screen while JS loads. React then "hydrates" the client components, making them interactive.
