Lesson 5: Where State Lives (Lifting State Up)

Each component owns its own state. But what happens when two sibling components need to share the same data?

The answer: move the state up to their common parent, and pass it down via props.


function Parent() {
  const [count, setCount] = useState(0);

  return (
    <>
      <Display count={count} />
      <Controls onIncrement={() => setCount(count + 1)} />
    </>
  );
}

function Display({ count }) {
  return <p>Count: {count}</p>;
}

function Controls({ onIncrement }) {
  return <button onClick={onIncrement}>Increment</button>;
}
Display and Controls don't talk to each other directly. The parent owns the state and passes down:

the value to whoever needs to read it
the setter (wrapped in a function) to whoever needs to change it
This is called lifting state up — it's a core React pattern you'll use constantly.

The rule of thumb: put state in the lowest common ancestor of the components that need it.

Lesson 6: useEffect

You know how React re-renders components when state changes? Sometimes you need to do something as a side effect of that — fetch data from an API, set a timer, update the document title. That's what useEffect is for.


import { useState, useEffect } from 'react';

function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => setUser(data));
  }, [userId]);

  if (!user) return <p>Loading...</p>;
  return <h1>{user.name}</h1>;
}
Breaking it down:

useEffect(fn, deps) — takes a function and a dependency array
React runs the function after the component renders
It re-runs only when values in the dependency array change
The dependency array controls when the effect fires:


useEffect(() => { ... });          // runs after EVERY render (rarely what you want)
useEffect(() => { ... }, []);      // runs ONCE, on mount only
useEffect(() => { ... }, [userId]); // runs on mount + whenever userId changes
The [] case is the equivalent of componentDidMount if you've seen older React code — it's how you trigger a one-time API call when the component first appears.

Lesson 7: Conditional Rendering & Lists

Conditional Rendering

Since JSX is just JavaScript, you use plain JS to show/hide things:


function Alert({ message, isError }) {
  return (
    <div>
      {isError && <p className="error">{message}</p>}
      {!isError && <p className="success">{message}</p>}
    </div>
  );
}
Or with a ternary for either/or:


{isError ? <p className="error">{message}</p> : <p className="success">{message}</p>}
Rendering Lists

To render an array of items, you use .map():


function UserList({ users }) {
  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
The key prop is required when rendering lists. React uses it internally to track which items changed, added, or removed — without it, React can't efficiently update the DOM. Always use a stable unique ID from your data, never the array index if the list can reorder.

These two patterns show up in almost every React component you'll ever write.
