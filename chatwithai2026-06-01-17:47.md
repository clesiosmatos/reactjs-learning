Lesson 16: State Management at Scale
When Context isn't enough
You already know Context API. It works great for auth, theme, user preferences — things that change rarely and are read by many components.

The problem shows up when:

State changes frequently (cart updates, real-time data, filters)
Multiple components need to write to the same state
You need derived state (e.g. cart total computed from items)
You want devtools to inspect state changes over time
Context re-renders every consumer on every change. For frequently-updating state, that's a performance problem. You also end up with deeply nested providers and scattered logic.

Option A: Zustand (recommended starting point)
Zustand is minimal — just a store and a hook. No providers, no boilerplate.


npm install zustand
Creating a store:


// store/cartStore.js
import { create } from 'zustand';

const useCartStore = create((set, get) => ({
  items: [],

  addItem: (product) =>
    set((state) => ({
      items: [...state.items, product],
    })),

  removeItem: (id) =>
    set((state) => ({
      items: state.items.filter((item) => item.id !== id),
    })),

  // Derived value — computed on the fly
  get total() {
    return get().items.reduce((sum, item) => sum + item.price, 0);
  },

  clearCart: () => set({ items: [] }),
}));

export default useCartStore;
Consuming it anywhere — no Provider needed:


function CartIcon() {
  const items = useCartStore((state) => state.items);
  return <span>{items.length} items</span>;
}

function ProductCard({ product }) {
  const addItem = useCartStore((state) => state.addItem);
  return <button onClick={() => addItem(product)}>Add to cart</button>;
}

function CartTotal() {
  const total = useCartStore((state) => state.total);
  return <p>Total: ${total}</p>;
}
The selector (state) => state.items is important — it means this component only re-renders when items changes, not when any other part of the store changes. That's the performance win Context can't give you.

Option B: Redux Toolkit (RTK)
Redux had a reputation for massive boilerplate. Redux Toolkit fixes that — it's the official modern way to use Redux.


npm install @reduxjs/toolkit react-redux
RTK introduces slices — a self-contained unit of state + reducers + actions:


// store/cartSlice.js
import { createSlice } from '@reduxjs/toolkit';

const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [] },
  reducers: {
    addItem: (state, action) => {
      state.items.push(action.payload); // RTK uses Immer — direct mutation is safe here
    },
    removeItem: (state, action) => {
      state.items = state.items.filter((item) => item.id !== action.payload);
    },
    clearCart: (state) => {
      state.items = [];
    },
  },
});

export const { addItem, removeItem, clearCart } = cartSlice.actions;
export default cartSlice.reducer;

// store/index.js
import { configureStore } from '@reduxjs/toolkit';
import cartReducer from './cartSlice';

export const store = configureStore({
  reducer: { cart: cartReducer },
});

// App.jsx — one Provider at the root
import { Provider } from 'react-redux';
import { store } from './store';

function App() {
  return (
    <Provider store={store}>
      <Router>...</Router>
    </Provider>
  );
}

// Consuming it
import { useSelector, useDispatch } from 'react-redux';
import { addItem } from './store/cartSlice';

function ProductCard({ product }) {
  const dispatch = useDispatch();
  return <button onClick={() => dispatch(addItem(product))}>Add to cart</button>;
}

function CartSummary() {
  const items = useSelector((state) => state.cart.items);
  const total = items.reduce((sum, item) => sum + item.price, 0);
  return <p>{items.length} items — ${total}</p>;
}
Zustand vs RTK — how to choose
Zustand	Redux Toolkit
Setup	Minimal, no Provider	Requires Provider + configureStore
Boilerplate	Very low	Low (much less than old Redux)
DevTools	Available	Excellent (Redux DevTools)
Team familiarity	Growing	Dominant in enterprise
Async (API calls)	Manual or middleware	RTK Query (built-in data fetching)
Learning curve	Low	Medium
Rule of thumb: new project or small team → Zustand. Existing Redux codebase or large team with strict patterns → RTK.

Lesson 17: Data Fetching — React Query / SWR
The naive approach and why it breaks down
You've seen useEffect + fetch. It works for simple cases:


function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(setUser)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [userId]);

  if (loading) return <Spinner />;
  if (error) return <p>Error</p>;
  return <p>{user.name}</p>;
}
This breaks down fast:

No caching — navigate away and back, it re-fetches
No deduplication — two components fetching the same endpoint make two requests
No background refresh — stale data sits until the component remounts
No race condition handling — if userId changes fast, older responses can overwrite newer ones
Lots of boilerplate — loading/error state in every component
React Query solves all of this.

React Query

npm install @tanstack/react-query
Setup — one Provider at the root:


// main.jsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';

const queryClient = new QueryClient();

ReactDOM.createRoot(document.getElementById('root')).render(
  <QueryClientProvider client={queryClient}>
    <App />
  </QueryClientProvider>
);
useQuery — fetching data

import { useQuery } from '@tanstack/react-query';

function UserProfile({ userId }) {
  const { data: user, isLoading, error } = useQuery({
    queryKey: ['user', userId],   // cache key
    queryFn: () => fetch(`/api/users/${userId}`).then(res => res.json()),
  });

  if (isLoading) return <Spinner />;
  if (error) return <p>Error: {error.message}</p>;
  return <p>{user.name}</p>;
}
queryKey is the cache identity. ['user', 42] and ['user', 43] are separate cache entries. If two components both call useQuery({ queryKey: ['user', 42] }), React Query makes one request and shares the result.

What you get for free

const { data, isLoading, isFetching, error, isStale, refetch } = useQuery({
  queryKey: ['user', userId],
  queryFn: fetchUser,
  staleTime: 60_000,        // data is fresh for 60s — no refetch during this window
  gcTime: 5 * 60_000,       // keep in cache for 5min after no subscribers
  retry: 3,                 // auto-retry on failure
  refetchOnWindowFocus: true, // refetch when tab regains focus (default: true)
});
isLoading — true on the very first fetch (no cached data yet)
isFetching — true whenever a fetch is in progress (including background refresh)
staleTime — the window where cached data is considered fresh, no refetch triggered
useMutation — writing data

import { useMutation, useQueryClient } from '@tanstack/react-query';

function UpdateNameForm({ userId }) {
  const queryClient = useQueryClient();

  const mutation = useMutation({
    mutationFn: (newName) =>
      fetch(`/api/users/${userId}`, {
        method: 'PATCH',
        body: JSON.stringify({ name: newName }),
      }).then(res => res.json()),

    onSuccess: () => {
      // Invalidate the cache — triggers a background refetch
      queryClient.invalidateQueries({ queryKey: ['user', userId] });
    },
  });

  return (
    <button
      onClick={() => mutation.mutate('New Name')}
      disabled={mutation.isPending}
    >
      {mutation.isPending ? 'Saving...' : 'Save'}
    </button>
  );
}
invalidateQueries marks the cache entry as stale and triggers a background refetch. Any component subscribed to ['user', userId] gets fresh data automatically.

Optimistic updates
Instead of waiting for the server, update the UI immediately and roll back on error:


const mutation = useMutation({
  mutationFn: updateUser,

  onMutate: async (newData) => {
    await queryClient.cancelQueries({ queryKey: ['user', userId] });

    const previous = queryClient.getQueryData(['user', userId]);
    queryClient.setQueryData(['user', userId], old => ({ ...old, ...newData }));

    return { previous }; // context passed to onError
  },

  onError: (err, newData, context) => {
    queryClient.setQueryData(['user', userId], context.previous); // rollback
  },

  onSettled: () => {
    queryClient.invalidateQueries({ queryKey: ['user', userId] }); // sync with server
  },
});
This is the same pattern you'd find in any optimistic UI — assume success, rollback on failure.

Dependent queries
Run a query only after another one completes:


function UserOrders({ userId }) {
  const { data: user } = useQuery({
    queryKey: ['user', userId],
    queryFn: () => fetchUser(userId),
  });

  const { data: orders } = useQuery({
    queryKey: ['orders', user?.id],
    queryFn: () => fetchOrders(user.id),
    enabled: !!user,   // don't run until user is loaded
  });
}
Pagination

const [page, setPage] = useState(1);

const { data, isPlaceholderData } = useQuery({
  queryKey: ['products', page],
  queryFn: () => fetchProducts(page),
  placeholderData: keepPreviousData, // keep showing page N while page N+1 loads

React Query vs SWR
SWR (from Vercel) is the lighter alternative:


import useSWR from 'swr';

const fetcher = (url) => fetch(url).then(res => res.json());

function UserProfile({ userId }) {
  const { data, error, isLoading } = useSWR(`/api/users/${userId}`, fetcher);
  // ...
}
React Query	SWR
Bundle size	~13kb	~4kb
Mutations	useMutation with full lifecycle	Manual
Devtools	Built-in	Limited
Pagination / infinite	Built-in	Basic
Optimistic updates	Built-in	Manual
Use when	Full-featured data layer	Simple fetching, smaller bundle
For most apps: React Query. For a Next.js app with simple needs: SWR.

The mental model
Think of React Query as a server state cache sitting between your components and your API — the same role Redis plays between your Node.js app and your database. It handles freshness, invalidation, deduplication, and background sync. Your components just declare what data they need.


Components → useQuery(key) → [cache hit? return it] → [stale? refetch in background]
                                                    ↓
                                              API call → update cache → components re-render

Lesson 18: Error Boundaries, Suspense & Lazy Loading
Error Boundaries
React renders a blank screen if an unhandled error is thrown during render. Error boundaries catch those errors and let you show a fallback UI instead — the same concept as a try/catch, but for the component tree.

The catch: error boundaries must be class components. There's no hook equivalent (yet). In practice, you write one once and reuse it everywhere.


// components/ErrorBoundary.jsx
import { Component } from 'react';

class ErrorBoundary extends Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, info) {
    // Log to your error tracker (Sentry, etc.)
    console.error(error, info.componentStack);
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback || <p>Something went wrong.</p>;
    }
    return this.props.children;
  }
}

export default ErrorBoundary;
Using it:


function App() {
  return (
    <ErrorBoundary fallback={<p>Failed to load dashboard.</p>}>
      <Dashboard />
    </ErrorBoundary>
  );
}
Place them strategically — not one at the very top (hides everything on any error), not one per component (too granular). Wrap logical sections: sidebar, main content, widgets.


function App() {
  return (
    <>
      <ErrorBoundary fallback={<p>Sidebar failed</p>}>
        <Sidebar />
      </ErrorBoundary>

      <ErrorBoundary fallback={<p>Main content failed</p>}>
        <MainContent />
      </ErrorBoundary>
    </>
  );
}
react-error-boundary — skip writing the class yourself:


npm install react-error-boundary

import { ErrorBoundary } from 'react-error-boundary';

function ErrorFallback({ error, resetErrorBoundary }) {
  return (
    <div>
      <p>Error: {error.message}</p>
      <button onClick={resetErrorBoundary}>Try again</button>
    </div>
  );
}

<ErrorBoundary FallbackComponent={ErrorFallback} onReset={() => refetch()}>
  <Dashboard />
</ErrorBoundary>
resetErrorBoundary clears the error state — pair it with onReset to also refetch data or reset any state that caused the error.

Suspense
Suspense lets a component "wait" for something before rendering — and show a fallback in the meantime. You've probably seen it with lazy loading, but it also works with data fetching (when your data library supports it).


import { Suspense } from 'react';

<Suspense fallback={<Spinner />}>
  <SlowComponent />
</Suspense>
React renders <Spinner /> until <SlowComponent /> is ready.

Lazy Loading — code splitting
By default, your entire app ships as one JS bundle. The user downloads code for every route, even ones they never visit. Lazy loading splits the bundle — each route loads its own chunk only when navigated to.


import { lazy, Suspense } from 'react';

// Instead of: import Dashboard from './pages/Dashboard'
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Settings = lazy(() => import('./pages/Settings'));
const Reports = lazy(() => import('./pages/Reports'));
import('./pages/Dashboard') is a dynamic import — it returns a Promise. React handles the loading state via Suspense.

With React Router:


import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

const Home = lazy(() => import('./pages/Home'));
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Settings = lazy(() => import('./pages/Settings'));

function App() {
  return (
    <Suspense fallback={<PageSpinner />}>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/dashboard" element={<Dashboard />} />
        <Route path="/settings" element={<Settings />} />
      </Routes>
    </Suspense>
  );
}
One <Suspense> wrapping all routes is enough — React shows the fallback whenever any lazy component is loading.

Combining Error Boundary + Suspense
They naturally compose — Suspense handles the loading state, ErrorBoundary handles the failure state:


<ErrorBoundary FallbackComponent={ErrorFallback}>
  <Suspense fallback={<Spinner />}>
    <Dashboard />
  </Suspense>
</ErrorBoundary>
Reading order: outer to inner on happy path, inner to outer on error:

Loading → Suspense shows <Spinner />
Loaded → renders <Dashboard />
Crash → ErrorBoundary shows <ErrorFallback />
Suspense with React Query
React Query supports Suspense mode — instead of handling isLoading manually, let Suspense do it:


// useQuery with suspense enabled
const { data } = useSuspenseQuery({
  queryKey: ['user', userId],
  queryFn: fetchUser,
});
// data is guaranteed non-null here — no isLoading check needed

// The component assumes data is ready
function UserProfile({ userId }) {
  const { data: user } = useSuspenseQuery({
    queryKey: ['user', userId],
    queryFn: () => fetchUser(userId),
  });

  return <p>{user.name}</p>; // no loading/error handling here
}

// Parent handles loading and errors declaratively
<ErrorBoundary FallbackComponent={ErrorFallback}>
  <Suspense fallback={<Spinner />}>
    <UserProfile userId={42} />
  </Suspense>
</ErrorBoundary>
This separates concerns cleanly: the component declares what data it needs, the parent declares how to handle loading and errors.

When to lazy load
Not everything needs lazy loading. The rule:

Candidate	Lazy load?
Route-level pages	Yes — always
Heavy third-party components (charts, editors, maps)	Yes
Modals / dialogs not shown on initial render	Yes
Small shared components (Button, Input)	No
Components always visible on first paint	No
