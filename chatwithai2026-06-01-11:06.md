Context API
Lesson 13: Context API — Global State
You already saw useContext in Lesson 8 as a way to avoid prop drilling. Now let's go deeper — how to actually architect it for real use cases like auth, theme, and cart state.

The problem it solves
Imagine this tree:


App
 └── Layout
      └── Navbar
           └── UserAvatar   ← needs `user`
Without context, you'd pass user as a prop through Layout and Navbar even though they don't use it — they're just middlemen. That's prop drilling.

The full pattern
Context has three parts: create, provide, consume.


// 1. Create — usually in its own file
import { createContext, useContext, useState } from 'react';

const AuthContext = createContext(null);

// 2. Provide — wrap the tree that needs access
export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);

  const login = async (credentials) => {
    const data = await api.login(credentials);
    setUser(data.user);
  };

  const logout = () => setUser(null);

  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

// 3. Consume — anywhere inside the Provider tree
export function useAuth() {
  return useContext(AuthContext);
}

// App.jsx — wire it up
function App() {
  return (
    <AuthProvider>
      <Router>
        <Routes>...</Routes>
      </Router>
    </AuthProvider>
  );
}

// UserAvatar.jsx — consume, no props needed
function UserAvatar() {
  const { user, logout } = useAuth();
  return <button onClick={logout}>{user.name}</button>;
}
The custom hook wrapper (useAuth)
Notice I exported useAuth() instead of exposing AuthContext directly. This is the standard pattern — for two reasons:

Cleaner API — consumers call useAuth() instead of the verbose useContext(AuthContext)
Guards against misuse — you can add a check:

export function useAuth() {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth must be used inside AuthProvider');
  return ctx;
}
If someone calls useAuth() outside the provider, they get a clear error instead of a silent null.

Real-world example: Theme context

const ThemeContext = createContext(null);

export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const toggle = () => setTheme(t => t === 'light' ? 'dark' : 'light');

  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  );
}

export const useTheme = () => useContext(ThemeContext);

function Header() {
  const { theme, toggle } = useTheme();
  return (
    <header className={theme}>
      <button onClick={toggle}>Switch to {theme === 'light' ? 'dark' : 'light'}</button>
    </header>
  );
}
Combining multiple contexts
You'll end up with several providers. The common pattern is to nest them in App.jsx:


function App() {
  return (
    <AuthProvider>
      <ThemeProvider>
        <CartProvider>
          <Router>...</Router>
        </CartProvider>
      </ThemeProvider>
    </AuthProvider>
  );
}
Each provider is independent. A component can consume any of them.

One important caveat: performance
Every component that calls useContext(SomeContext) re-renders when the context value changes. If you put too much in one context, a small change (like updating a notification count) causes the entire tree consuming that context to re-render.

Solution: split contexts by how frequently they change:

Context	Changes
AuthContext	Rarely (login/logout)
ThemeContext	Occasionally
CartContext	Frequently
Don't put everything in one global context object.

useRef, lets go
Lesson 14: useRef
useRef serves two completely different purposes, and understanding both is key.

Purpose 1: Access a DOM element directly
Sometimes you need to reach into the DOM — focus an input, measure an element's size, trigger a video to play. useRef gives you a direct reference to the DOM node.


import { useRef } from 'react';

function SearchBar() {
  const inputRef = useRef(null);

  const handleButtonClick = () => {
    inputRef.current.focus(); // direct DOM call
  };

  return (
    <>
      <input ref={inputRef} type="text" placeholder="Search..." />
      <button onClick={handleButtonClick}>Focus input</button>
    </>
  );
}
ref={inputRef} tells React: "when this DOM element is created, put it in inputRef.current." After mount, inputRef.current is the actual DOM node — same as document.querySelector('input').

Common use cases:

Auto-focus an input on mount
Scroll an element into view
Read an element's dimensions (getBoundingClientRect)
Control a <video> or <canvas>
Purpose 2: Persist a value across renders WITHOUT triggering re-render
This is the one people miss. useRef is like useState but changing it doesn't cause a re-render.


function Timer() {
  const [seconds, setSeconds] = useState(0);
  const intervalRef = useRef(null); // stores the interval ID

  const start = () => {
    intervalRef.current = setInterval(() => {
      setSeconds(s => s + 1);
    }, 1000);
  };

  const stop = () => {
    clearInterval(intervalRef.current); // still accessible, no re-render needed
  };

  return (
    <>
      <p>{seconds}s</p>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
    </>
  );
}
Why not useState for the interval ID? Because you don't need the UI to update when it changes — you just need to store it somewhere that survives re-renders so you can call clearInterval later.

The key distinction
useState	useRef
Survives re-renders	✓	✓
Triggers re-render on change	✓	✗
Use for	UI data	DOM refs, timers, prev values
Practical pattern: tracking previous value
A classic use of useRef — knowing what a value was on the previous render:


function PriceDisplay({ price }) {
  const prevPriceRef = useRef(null);

  useEffect(() => {
    prevPriceRef.current = price; // update AFTER render
  });

  const prevPrice = prevPriceRef.current;

  return (
    <p style={{ color: price > prevPrice ? 'green' : 'red' }}>
      {price} {prevPrice !== null && `(was ${prevPrice})`}
    </p>
  );
}
React updates prevPriceRef.current after each render, so during the current render it still holds the previous value.

The rule
useRef does not notify React when .current changes. So:


// This will NOT update the UI
const countRef = useRef(0);
countRef.current += 1; // React has no idea this happened
If you need the UI to reflect the change → useState. If you just need to store something → useRef.

Let's go Forms (controlled inputs, react-hook-form)
Lesson 15: Forms
Part 1: Controlled Inputs
In plain HTML, form elements manage their own state internally. In React, you manage it — the input value is always driven by React state. This is called a controlled input.


function LoginForm() {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');

  const handleSubmit = (e) => {
    e.preventDefault(); // prevent page reload
    console.log({ email, password });
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={email}
        onChange={e => setEmail(e.target.value)}
        type="email"
      />
      <input
        value={password}
        onChange={e => setPassword(e.target.value)}
        type="password"
      />
      <button type="submit">Login</button>
    </form>
  );
}
The flow: user types → onChange fires → state updates → React re-renders with new value. The input always reflects state — it can never get out of sync with your data.

The problem with this approach at scale
For a form with 10 fields + validation + error messages + submission state, the useState approach explodes:


// This gets out of hand fast
const [name, setName] = useState('');
const [email, setEmail] = useState('');
const [phone, setPhone] = useState('');
const [nameError, setNameError] = useState('');
const [emailError, setEmailError] = useState('');
// ... and so on
This is exactly where useReducer helps (as you saw in Lesson 10) — but even that requires writing a lot of boilerplate. That's why react-hook-form exists.

Part 2: react-hook-form

npm install react-hook-form
The key insight: react-hook-form uses uncontrolled inputs under the hood (refs, not state), which means no re-render on every keystroke. It only re-renders when necessary — on submit or when validation fires.

Basic usage

import { useForm } from 'react-hook-form';

function LoginForm() {
  const {
    register,
    handleSubmit,
    formState: { errors }
  } = useForm();

  const onSubmit = (data) => {
    console.log(data); // { email: '...', password: '...' }
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input
        {...register('email', {
          required: 'Email is required',
          pattern: { value: /\S+@\S+\.\S+/, message: 'Invalid email' }
        })}
        type="email"
      />
      {errors.email && <p>{errors.email.message}</p>}

      <input
        {...register('password', {
          required: 'Password is required',
          minLength: { value: 8, message: 'Min 8 characters' }
        })}
        type="password"
      />
      {errors.password && <p>{errors.password.message}</p>}

      <button type="submit">Login</button>
    </form>
  );
}
register('fieldName', rules) returns props you spread onto the input — it wires up ref, onChange, onBlur, and name automatically.

Validation rules built in

register('age', {
  required: 'Required',
  min: { value: 18, message: 'Must be 18+' },
  max: { value: 99, message: 'Must be under 100' },
  pattern: { value: /^[0-9]+$/, message: 'Numbers only' },
  validate: value => value % 2 === 0 || 'Must be even' // custom function
})
Watching field values
Sometimes you need to react to a field's value — e.g. show a second field only if the first has a certain value:


const { register, watch } = useForm();
const accountType = watch('accountType');

return (
  <>
    <select {...register('accountType')}>
      <option value="personal">Personal</option>
      <option value="business">Business</option>
    </select>

    {accountType === 'business' && (
      <input {...register('companyName', { required: true })} placeholder="Company name" />
    )}
  </>
);
Submission states

const { handleSubmit, formState: { isSubmitting, isSubmitSuccessful } } = useForm();

const onSubmit = async (data) => {
  await api.login(data); // react-hook-form sets isSubmitting=true during this
};

return (
  <form onSubmit={handleSubmit(onSubmit)}>
    <button type="submit" disabled={isSubmitting}>
      {isSubmitting ? 'Logging in...' : 'Login'}
    </button>
    {isSubmitSuccessful && <p>Success!</p>}
  </form>
);
Setting server-side errors
When the API returns a validation error (e.g. "email already taken"):


const { setError } = useForm();

const onSubmit = async (data) => {
  try {
    await api.register(data);
  } catch (e) {
    setError('email', { message: 'Email already in use' });
  }
};
Controlled vs react-hook-form — when to use each
Situation	Use
1–3 fields, simple form	useState controlled inputs
4+ fields with validation	react-hook-form
Complex dependent fields	react-hook-form + watch
Need real-time UI sync (e.g. live preview)	Controlled inputs or watch
Performance matters (large forms)	react-hook-form (uncontrolled)
