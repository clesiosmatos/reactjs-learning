React 18 concurrent features (useTransition, useDeferredValue)
Question 27 — React 18 Concurrent Features

The mental model first

React 18 introduced the concept of concurrent rendering — React can now pause, interrupt, or abandon a render mid-way to handle something more urgent.

Think of it like Node.js's event loop: long synchronous work blocks everything. React 18 lets you mark some state updates as "low priority" so the browser stays responsive while React processes them in the background.

Before React 18, every state update was treated equally — urgent and non-urgent updates were in the same queue.

useTransition — mark an update as non-urgent

Classic problem: a search input that filters a huge list. Every keystroke triggers a filter over 10,000 items — the input feels laggy.


import { useState, useTransition } from 'react';

function SearchPage({ items }: { items: string[] }) {
  const [query, setQuery]   = useState('');
  const [filtered, setFiltered] = useState(items);
  const [isPending, startTransition] = useTransition();

  function handleChange(e: React.ChangeEvent<HTMLInputElement>) {
    const value = e.target.value;

    // Urgent: update the input immediately
    setQuery(value);

    // Non-urgent: defer the expensive filter
    startTransition(() => {
      setFiltered(items.filter(i => i.includes(value)));
    });
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      {isPending && <Spinner />}  {/* React is still processing */}
      <List items={filtered} />
    </>
  );
}
What React does: it renders the input update immediately (so typing feels instant), then processes the filter in the background. If you type again before the filter finishes, React throws away the stale render and starts fresh — no wasted work.

isPending is true while the transition is in progress — useful for showing a loading indicator without blocking the UI.

useDeferredValue — defer a value, not an update

Same idea, but for when you don't control the state setter — e.g., the value comes from props or a parent:


import { useState, useDeferredValue, memo } from 'react';

const HeavyList = memo(function HeavyList({ query }: { query: string }) {
  // Expensive render — filters 10k items
  const filtered = items.filter(i => i.includes(query));
  return <ul>{filtered.map(i => <li key={i}>{i}</li>)}</ul>;
});

function SearchPage() {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);
  // deferredQuery lags behind query — it's the "last stable value"

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      {query !== deferredQuery && <Spinner />}  {/* stale while updating */}
      <HeavyList query={deferredQuery} />
    </>
  );
}
deferredQuery starts equal to query. When query changes, deferredQuery stays on the old value while React tries to re-render HeavyList in the background. Once it finishes, deferredQuery catches up.

When to use which

useTransition	useDeferredValue
You control the state setter	Yes	No (value from props/parent)
Get isPending flag	Yes	No (compare value !== deferred manually)
Use case	Button clicks, search inputs	Child component receiving a prop
Node.js analogy

startTransition is like setImmediate() vs synchronous code — you're telling React "do this after the urgent stuff." The difference is React can also cancel the deferred work if something more urgent comes in, which setImmediate can't do.

What seniors know that juniors miss

These APIs only help when the render itself is slow (large lists, heavy computation). If the bottleneck is a network request, use React Query's loading states instead.
useDeferredValue requires the deferred component to be wrapped in React.memo — otherwise the deferred value changes but the component re-renders anyway, defeating the purpose.
Both require React 18+. They're no-ops in React 17 (no error, just no benefit).

Advanced patterns — compound components, render props, HOCs
Question 28 — Advanced Patterns

These are pattern-recognition topics — seniors are expected to read code using them and know when (and when not) to apply them.

1. Higher-Order Components (HOCs)

A function that takes a component and returns a new component with added behavior. The OG pattern from pre-hooks React — you still see it in legacy codebases.


// HOC: wraps any component with auth protection
function withAuth<T extends object>(Component: React.ComponentType<T>) {
  return function AuthGuard(props: T) {
    const { isAuthenticated } = useAuth();

    if (!isAuthenticated) return <Redirect to="/login" />;
    return <Component {...props} />;
  };
}

// Usage
const ProtectedDashboard = withAuth(Dashboard);
const ProtectedSettings  = withAuth(Settings);
Node.js analogy: Express middleware. app.use(authMiddleware) wraps every route handler — same idea, different syntax.

When you still see HOCs today:

React.memo is technically a HOC
connect() from Redux (legacy)
Third-party libraries that haven't migrated to hooks
Why hooks replaced most HOCs: HOCs cause "wrapper hell" in DevTools — withAuth(withLogging(withTheme(Dashboard))) shows 3 extra layers. A custom hook does the same with zero extra nodes.

2. Render Props

Pass a function as a prop — the component calls it to render its output. Gives the parent full control over what gets rendered.


// Component exposes its state via a render prop
function MouseTracker({ render }: { render: (pos: { x: number; y: number }) => React.ReactNode }) {
  const [pos, setPos] = useState({ x: 0, y: 0 });

  return (
    <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}>
      {render(pos)}  {/* caller decides what to do with pos */}
    </div>
  );
}

// Usage — two completely different UIs, same tracking logic
<MouseTracker render={({ x, y }) => <p>Mouse at {x}, {y}</p>} />
<MouseTracker render={({ x, y }) => <Circle cx={x} cy={y} />} />
The children prop variant is the same pattern, just cleaner syntax:


function MouseTracker({ children }: { children: (pos: { x: number; y: number }) => React.ReactNode }) {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  return (
    <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}>
      {children(pos)}
    </div>
  );
}

// Usage
<MouseTracker>
  {({ x, y }) => <p>Mouse at {x}, {y}</p>}
</MouseTracker>
Why hooks replaced most render props: useMousePosition() custom hook does this with less syntax noise. But render props still appear in libraries (React Final Form, Downshift, Headless UI).

3. Compound Components — the one that's still widely used

A set of components that share implicit state through context. They look like HTML elements (<select> + <option>) — they only make sense together.


// The shared context
const TabsContext = React.createContext<{
  active: string;
  setActive: (id: string) => void;
} | null>(null);

// Root component — owns the state
function Tabs({ children, defaultTab }: { children: React.ReactNode; defaultTab: string }) {
  const [active, setActive] = useState(defaultTab);
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      <div className="tabs">{children}</div>
    </TabsContext.Provider>
  );
}

// Sub-components — consume the shared state
function TabList({ children }: { children: React.ReactNode }) {
  return <div role="tablist">{children}</div>;
}

function Tab({ id, children }: { id: string; children: React.ReactNode }) {
  const { active, setActive } = useContext(TabsContext)!;
  return (
    <button
      role="tab"
      aria-selected={active === id}
      onClick={() => setActive(id)}
    >
      {children}
    </button>
  );
}

function TabPanel({ id, children }: { id: string; children: React.ReactNode }) {
  const { active } = useContext(TabsContext)!;
  if (active !== id) return null;
  return <div role="tabpanel">{children}</div>;
}

// Attach sub-components to the root
Tabs.List  = TabList;
Tabs.Tab   = Tab;
Tabs.Panel = TabPanel;
Usage is expressive and flexible:


<Tabs defaultTab="profile">
  <Tabs.List>
    <Tabs.Tab id="profile">Profile</Tabs.Tab>
    <Tabs.Tab id="settings">Settings</Tabs.Tab>
  </Tabs.List>

  <Tabs.Panel id="profile"><ProfileForm /></Tabs.Panel>
  <Tabs.Panel id="settings"><SettingsForm /></Tabs.Panel>
</Tabs>
Why this pattern matters: the consumer controls the structure (where tabs go, what panels contain) without the <Tabs> component needing 15 props to handle every layout variation. Headless UI, Radix UI, and Reach UI are all built on this pattern.

Pattern decision tree


Need to reuse logic across components?
  └─ Logic only (no UI)           → Custom hook  ✓ (first choice)
  └─ Logic + wrapping behavior    → HOC  (rare today)
  └─ Logic + caller controls UI   → Render prop  (mostly in libraries)

Need flexible, composable UI components?
  └─ Components share state       → Compound components  ✓
