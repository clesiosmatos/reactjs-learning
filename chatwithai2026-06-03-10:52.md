Dynamic routes and generateStaticParams
[slug] catches any value. For SSG, tell Next.js which values to pre-render:


// app/blog/[slug]/page.tsx

// Runs at build time — generates /blog/post-1, /blog/post-2, etc.
export async function generateStaticParams() {
  const posts = await fetch('https://api.example.com/posts').then(r => r.json());
  return posts.map(post => ({ slug: post.slug }));
}

// Runs at build time for each slug
async function BlogPost({ params }: { params: { slug: string } }) {
  const post = await fetch(`https://api.example.com/posts/${params.slug}`)
    .then(r => r.json());
  return <article>{post.title}</article>;
}
Without generateStaticParams, the page renders on-demand (SSR). With it, pages are pre-built at deploy time (SSG).

import { NextRequest, NextResponse } from 'next/server';

Question 25 — React Performance: the mental model

Before touching any API, the core insight:

React re-renders a component every time its state or props change. A re-render is cheap — it's just a function call. The problem is cascading re-renders: a parent re-renders, all its children re-render, their children re-render, etc. — even if nothing relevant changed for them.

Think of it like a Node.js event emitter where one emit('change') triggers 50 listeners — most of which don't care about that specific change.

Profiling first — never optimize blind
React DevTools has a Profiler tab. You record an interaction and it shows you:

Which components re-rendered
Why they re-rendered ("props changed", "state changed", "parent re-rendered")
How long each render took

Install: React Developer Tools browser extension
Open DevTools → "Profiler" tab → Record → click something → Stop
The flame graph shows render time per component. You look for wide bars (slow renders) or components re-rendering that shouldn't be.

React.memo — skip re-renders when props didn't change

// Without memo: re-renders every time parent renders, even if item didn't change
function ProductCard({ item }: { item: Product }) {
  return <div>{item.name}</div>;
}

// With memo: skips re-render if item reference is the same
const ProductCard = React.memo(function ProductCard({ item }: { item: Product }) {
  return <div>{item.name}</div>;
});
Node.js analogy: like caching a computed result — if the input didn't change, return the cached output.

Catch: memo does a shallow comparison. If the parent creates a new object/array/function on every render, memo won't help — the reference changes even if the data is identical.


// This breaks memo — new array reference on every parent render
<ProductCard items={products.filter(p => p.active)} />

// Fix: move the filter outside, or use useMemo
const activeProducts = useMemo(() => products.filter(p => p.active), [products]);
<ProductCard items={activeProducts} />
useMemo — memoize expensive calculations

function ProductList({ products, searchTerm }: Props) {
  // Without useMemo: recalculates on every render
  const filtered = products.filter(p => p.name.includes(searchTerm));

  // With useMemo: only recalculates when products or searchTerm changes
  const filtered = useMemo(
    () => products.filter(p => p.name.includes(searchTerm)),
    [products, searchTerm]
  );

  return filtered.map(p => <ProductCard key={p.id} item={p} />);
}
Rule of thumb: only useMemo when the calculation is genuinely expensive (sorting/filtering large arrays, heavy math) or when the result is passed as a prop to a memoized child.

useCallback — stable function references

function Parent() {
  const [count, setCount] = useState(0);

  // New function reference on every render → breaks memo on Child
  const handleClick = () => console.log('clicked');

  // Stable reference → memo on Child works
  const handleClick = useCallback(() => console.log('clicked'), []);

  return <Child onClick={handleClick} />;
}
useCallback(fn, deps) is just useMemo(() => fn, deps) — same mechanism, syntactic sugar for functions.

Lazy loading — don't ship what the user hasn't asked for yet
This is code splitting at the component level. Same concept as dynamic require() or import() in Node.js:


import { lazy, Suspense } from 'react';

// Bundle is NOT downloaded until this component is first rendered
const HeavyChart = lazy(() => import('./HeavyChart'));
const AdminPanel = lazy(() => import('./AdminPanel'));

function App() {
  const [showChart, setShowChart] = useState(false);

  return (
    <div>
      <button onClick={() => setShowChart(true)}>Show Chart</button>

      {showChart && (
        <Suspense fallback={<Spinner />}>
          <HeavyChart />  {/* JS downloaded here, on demand */}
        </Suspense>
      )}
    </div>
  );
}
The bundler (Vite/webpack) splits HeavyChart into a separate .js chunk. It's only downloaded when the user triggers it.

Route-level code splitting — the most impactful pattern
Most apps lazy-load entire routes. Users only download the JS for pages they visit:


import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

const Home      = lazy(() => import('./pages/Home'));
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Settings  = lazy(() => import('./pages/Settings'));

function App() {
  return (
    <Suspense fallback={<PageSpinner />}>
      <Routes>
        <Route path="/"          element={<Home />} />
        <Route path="/dashboard" element={<Dashboard />} />
        <Route path="/settings"  element={<Settings />} />
      </Routes>
    </Suspense>
  );
}
Before: one 800KB bundle, user downloads everything on first load.
After: 150KB initial bundle + chunks per route, downloaded on navigation.

The decision tree

Component re-rendering too often?
  └─ Is it slow to render?         → React.memo
  └─ Does it receive functions?    → useCallback on those functions
  └─ Does it receive objects/arrays? → useMemo on those values

Initial load too slow?
  └─ Large components not needed immediately → React.lazy + Suspense
  └─ Route-based → lazy load each page route

Question 26 — Testing: the mental model

First, the philosophy seniors are expected to know:

Test behavior, not implementation. A user doesn't care which hook you used — they care that clicking "Submit" sends the form data.

This comes from Kent C. Dodds (the RTL author):

"The more your tests resemble the way your software is used, the more confidence they give you."

The testing pyramid for React

        /\
       /  \   E2E (Playwright, Cypress)
      /----\  — full browser, slow, test critical paths only
     /      \
    /--------\ Integration (React Testing Library)
   / ★ most   \ — render component tree, interact, assert DOM
  /------------\
 /              \ Unit (pure functions, hooks)
/----------------\ — utils, custom hooks in isolation
Seniors spend most time in the middle layer — RTL integration tests. Pure unit tests for utilities, E2E for critical flows (checkout, login).

React Testing Library — core concept
RTL renders your component into a real DOM (via jsdom) and gives you queries that mirror how a user finds elements:


import { render, screen, fireEvent, waitFor } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import Counter from './Counter';

test('increments count when button clicked', async () => {
  const user = userEvent.setup();

  render(<Counter />);

  // Query by what the user sees — text, role, label
  expect(screen.getByText('Count: 0')).toBeInTheDocument();

  await user.click(screen.getByRole('button', { name: /increment/i }));

  expect(screen.getByText('Count: 1')).toBeInTheDocument();
});
Key point: getByRole('button', { name: /increment/i }) — this is how a screen reader finds the button. If this query fails, your a11y is broken too.

Query priority — which to use when

getByRole          ← first choice, mirrors a11y tree
getByLabelText     ← for form inputs (finds by <label>)
getByPlaceholderText ← acceptable fallback for inputs
getByText          ← for non-interactive text
getByTestId        ← last resort (data-testid="x"), means no semantic HTML
Seniors know: if you're reaching for getByTestId everywhere, it's a smell — your HTML isn't semantic.

Async testing — the common pitfall

// Testing a component that fetches data
function UserProfile({ id }: { id: string }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetch(`/api/users/${id}`).then(r => r.json()).then(setUser);
  }, [id]);

  if (!user) return <p>Loading...</p>;
  return <h1>{user.name}</h1>;
}

test('shows user name after loading', async () => {
  // Mock fetch
  global.fetch = jest.fn().mockResolvedValue({
    json: () => Promise.resolve({ name: 'Alice' }),
  });

  render(<UserProfile id="1" />);

  // Wrong: getBy throws immediately if not found
  // expect(screen.getByText('Alice')).toBeInTheDocument(); ❌

  // Right: waitFor retries until the assertion passes or times out
  await waitFor(() => {
    expect(screen.getByText('Alice')).toBeInTheDocument();
  });

  // Or shorter: findBy = getBy + waitFor combined
  expect(await screen.findByText('Alice')).toBeInTheDocument(); ✓
});
Rule: getBy = synchronous, throws immediately. findBy = async, waits. queryBy = synchronous, returns null instead of throwing (use for asserting absence).

Mocking — what seniors actually do
Mock external dependencies, not internals:


// Mock an entire module
jest.mock('../api/users', () => ({
  fetchUser: jest.fn().mockResolvedValue({ name: 'Alice' }),
}));

// Mock just one function
import { fetchUser } from '../api/users';
jest.spyOn(fetchUser, 'mockResolvedValueOnce')({ name: 'Alice' });
With React Query (most real apps):


import { QueryClient, QueryClientProvider } from '@tanstack/react-query';

function renderWithQuery(ui: React.ReactElement) {
  const queryClient = new QueryClient({
    defaultOptions: { queries: { retry: false } }, // don't retry on test failures
  });
  return render(
    <QueryClientProvider client={queryClient}>{ui}</QueryClientProvider>
  );
}

test('shows users list', async () => {
  server.use(
    http.get('/api/users', () => HttpResponse.json([{ name: 'Alice' }]))
  );

  renderWithQuery(<UserList />);

  expect(await screen.findByText('Alice')).toBeInTheDocument();
});
The pattern here is MSW (Mock Service Worker) — intercepts actual HTTP requests at the network level. More reliable than mocking fetch manually, because your component code is unchanged.

What seniors test vs don't test
Test:

User interactions (click, type, submit)
Conditional rendering (shows error state, loading state, empty state)
Integration between components (form submits and updates list)
Accessibility (queries via role/label enforce it)
Don't test:

Implementation details (useState value directly, internal function calls)
Third-party library behavior (React Router already tested <Link>)
Snapshot tests for complex components (they catch everything, signal nothing)
Snapshot testing — the nuance

// Fine for small, stable UI components
test('Button renders correctly', () => {
  const { container } = render(<Button variant="primary">Click</Button>);
  expect(container).toMatchSnapshot();
});
Seniors know: snapshot tests fail on any change — including intentional ones. They become noise. Use them sparingly for leaf UI components (design system buttons, badges), never for page-level components.
